## §1. 9월 위기 — 첫 코딩 후 추정보다 시간 소요

### 증상 체크

다음 중 2개 이상 해당하면 위기:
- [ ] 9월 둘째 주 끝에 채팅 탭의 5단계 번역이 작동하지 않는다
- [ ] React 또는 Tailwind 학습 시간이 예상보다 50% 이상 더 걸렸다
- [ ] Supabase RLS 정책 충돌로 1주일 이상 막혔다
- [ ] Edge Function 디버깅에서 매번 30분 이상 소요된다

### 원인 진단

가장 흔한 원인 3가지:

1. **AI 페어 코딩 의존도 과다** — Claude Code 등에 의존하다 환각 코드 디버깅에 시간 소요
2. **TypeScript strict 모드 + Zod 학습 곡선** — 처음 사용 시 타입 에러 해결에 시간 소요
3. **로컬 환경 문제** — Supabase 로컬 실행, env var 등

### 대응 옵션

**옵션 A — 5단계 번역 단순화**
5단계 모두 한 번에 구현하지 말고 우선순위로 나누기:
- 9월: STEP 0 (문화 교정) + STEP 2 (번역 3 버전) — 핵심 차별점만
- 10월: STEP 1 (구조 대응표) + STEP 3 (문법) — 학습 가치
- 11월: STEP 4 (문화 맥락) + STEP 5 (니모닉) — 부가 가치

**옵션 B — 라이브러리 선택 단순화**
- Zod → 단순 TypeScript 타입만 (검증 런타임 생략)
- TypeScript strict → 일단 strict: false로 시작, 출시 직전 strict 적용
- Tailwind → 익숙한 CSS 모듈로 전환

**옵션 C — 외부 도움 받기**
- ChatGPT/Claude에 문제 상황 자세히 설명하고 디버깅 조언
- Stack Overflow 또는 Reddit 관련 서브에 질문

### 결정 후 다음 단계

옵션 결정 후 즉시:
1. 일정 재조정 — 영향 받는 마일스톤을 README에 업데이트
2. PRD에 변경 기록 — 7월에 결정한 사항이 9월에 변경된 이유 명시
3. 다음 1주일 작업을 "작은 승리" 단위로 쪼개기 (오늘 끝낼 수 있는 것)

## §2. 10월 위기 — 에피소드 작성 지지부진

### 증상 체크

- [ ] 10월 둘째 주에 EP-01 초안이 완성 안 됨
- [ ] 자료 조사에서 1편당 4시간 이상 소요
- [ ] 할머니 말투 작성이 어색하게 느껴진다
- [ ] 사실 검증에서 막혀서 진도가 안 나간다

### 원인 진단

EPISODE_PRODUCTION_GUIDE의 추정(편당 220-350분)을 초과하는 가장 흔한 이유:

1. **자료 조사의 깊이 욕심** — 위키 1개로 충분한데 5개 cross-check 시도
2. **완벽주의** — 첫 초안에서 출판 수준 글쓰기 목표
3. **할머니 말투의 일관성 고민** — 이미 EP-04 모범 사례에 패턴이 있음

### 대응 옵션

**옵션 A — EP-04를 그대로 복제 후 변형**
- EP-04의 5단계 구조를 그대로 베끼고 내용만 교체
- "처음부터 새로 쓴다"가 아니라 "기존 템플릿 변형"
- 작성 시간 60% 절감 가능

**옵션 B — Tier 1 8편 → 5편으로 축소**
- 최우선 5편: お正月 / 入園式 / 飲み会 / 在留카드 / 회람판(EP-04)
- 나머지 3편(ふるさと納税 / お盆 / 七五三)은 출시 후 매월 1편씩 추가

**옵션 C — 외부 위탁 1-2편**
- 일본 거주 한국인 라이터 또는 일본어 강사
- 편당 ¥50,000~¥80,000
- 본인 작성 5편 + 위탁 3편으로 분담

### 결정 후 다음 단계

옵션 A 권장 (가장 빠르고 비용 0). 1주일 안에 EP-01 완성 못하면 옵션 B로 전환.

## §3. 10월 위기 — 베타 리뷰어 응답 지연

### 증상 체크

- [ ] EP-01을 보낸 지 5일 이상 지났는데 응답 없음
- [ ] 3명 중 1명만 응답
- [ ] 응답을 받았으나 "좋아요" 정도의 짧은 피드백

### 원인 진단

베타 리뷰어 단계에서 발생하는 흔한 문제:

1. **부담스러운 요청량** — 한 번에 5편을 보냄 → 부담
2. **체크리스트 양식 미사용** — "그냥 읽고 의견 주세요" → 답하기 어려움
3. **보상 명시 부족** — "선물 드릴게요"가 추상적

### 대응 옵션

**옵션 A — 1편씩, 체크리스트 명시**
- 1편씩만 발송, 체크리스트 양식 함께 전달 (EPISODE_PRODUCTION_GUIDE 참조)
- 응답 데드라인 명시 ("1주일 내 답주시면 좋겠습니다")
- 응답 시 체크리스트의 ★/★★/★★★ 평가 부탁

**옵션 B — 보상 구체화**
- "선물 드릴게요" → "이번 주말 신주쿠 한식당 식사 (¥3,000 상당) 어떠세요?"
- 누적 보상 명시 (¥10,000~¥15,000)

**옵션 C — 응답 안 한 사람 교체**
- 7일 응답 없으면 다음 사람에게 (1차 섭외 시 3명 확보의 이유)
- 일한모 게시판에 추가 모집 글 (응답률 30% 가정 시 5명 후보 필요)

### 결정 후 다음 단계

A + B를 동시에 진행. 그래도 응답 없으면 C로 전환.

## §4. 11월 위기 — 동기 저하 ("이걸 왜 하고 있지?")

### 증상 체크

가장 위험한 위기. 한 번에 결과가 나와서 알아차리기 어려움.

- [ ] 주말에 작업 안 하고 다른 것 함
- [ ] "포트폴리오니까 안 끝내도 돼"라는 생각
- [ ] PRD나 README를 다시 보지 않음
- [ ] 작은 변경에도 의욕이 안 남
- [ ] "이거 출시해도 사용자 없을 텐데"라는 합리화

### 원인 진단 — 동기 저하의 진짜 원인

표면 원인이 아닌 구조적 원인을 봐야 함:

| 표면 | 진짜 |
|---|---|
| "시간 부족" | "이게 가치 있는지 확신 없음" |
| "기술적 문제" | "끝낸 후 보여줄 사람 없음" |
| "본업 바쁨" | "본업이 더 의미 있게 느껴짐" |
| "콘텐츠 막힘" | "내가 정말 일본어 학습자에게 도움이 되나" 의문 |

### 대응 옵션

**옵션 A — "왜?"를 다시 쓰기**
- 7월에 작성한 README의 "프로젝트 컨텍스트" 섹션 다시 읽기
- 그 외에 본인이 5월에 작성한 PRD 5.0 §1, §2 다시 읽기
- "포트폴리오 검증 무대"라는 정체성을 다시 명시

**옵션 B — 작은 사용자 만들기**
- 베타 리뷰어 3명에게 "지금까지 만든 것 보여드릴게요" 미니 데모
- 그들의 짧은 반응이 강력한 동기
- 미완성이어도 OK — 진짜 사용자에게 보여주는 게 핵심

**옵션 C — 출시 시점 미루기**
- "12월 출시"가 동기 압박이라면, 1월 말로 미루는 결정
- README에 새 일정 기록 → 압박 해소 → 동기 회복
- 미룸 자체가 "포트폴리오 검증 무대" 정체성과 모순 안 됨

**옵션 D — 일단 멈추고 회고**
- 1주일 멈추기. 그동안 일본어 학습자 1명을 만나서 인터뷰 (대면 또는 Zoom)
- "내가 만들고 있는 것이 정말 도움이 되나"를 그 사람의 입에서 듣기
- 인터뷰 후 결정: 계속할지, 정체성 재조정할지, 멈출지

### 결정 후 다음 단계

**가장 중요**: 옵션 D를 먼저 시도. 인터뷰 30분이 다른 어떤 옵션보다 강력. 그 후 A/B/C 결정.

## §5. 11월 위기 — Eval 정확도 저하 (회귀)

### 증상 체크

- [ ] 7월 첫 실행 시 80% 통과했던 binary accuracy가 11월 재실행 시 70%로 떨어짐
- [ ] 새로 추가한 케이스에서 type accuracy 30% 미달
- [ ] 사용자 피드백에서 "AI가 한국어 그대로 직역해요" 보고

### 원인 진단

회귀 발생 흐름:

1. **Anthropic 모델 업데이트** — Haiku 4.5 → 4.6 자동 전환
2. **프롬프트 점진 변경** — 다른 의도로 수정한 부분이 영향
3. **새 케이스의 약점 노출** — 첫 90개에 없던 패턴 발견

### 대응 옵션

**옵션 A — 모델 버전 고정**
- API 호출 시 모델 명시: `claude-haiku-4-5-20251001` 같은 정확한 버전
- 자동 업데이트 의존하지 말고 명시적 핀

**옵션 B — 회귀 테스트 자동화**
- GitHub Actions로 매주 1회 자동 실행
- 결과가 임계값 미달 시 알림
- v1.2 단계 목표가 이미 EVAL_README에 있으므로 그대로 사용

**옵션 C — 데이터셋 확장**
- 사용자 피드백에서 발견한 새 패턴을 골든 데이터셋에 추가
- 90 → 120 케이스로 확장
- v3 프롬프트로 재실행

### 결정 후 다음 단계

A를 즉시 적용 (코드 한 줄). B는 다음 주말. C는 12월에.

## §6. 12월 위기 — 출시 시점 미루기

### 증상 체크

- [ ] 12월 중반인데 베타 5명도 시작 못함
- [ ] 변호사 검토 받지 못함
- [ ] 마지막 1-2개 기능이 안 끝남
- [ ] "출시했다가 망신만 당할 거 같다"

### 원인 진단

이건 위기지만 **합리적 결정의 기회**일 수 있음.

7월 → 12월 6개월 1인 작업의 현실:
- 추정 시간 × 1.3~1.5 = 실제 시간 (소프트웨어 산업 평균)
- 본업 + 본 프로젝트 병행 시 추가 0.2x
- 12월 후반 = 연말 휴가로 베타 응답률 30% 하락

### 대응 옵션

**옵션 A — 1월 말로 미루기 (강력 권장)**
- 12월 출시 강행보다 1월 말이 시장 부적합도 낮음
- 1월 첫 주: 신년 마케팅 시즌 (앱스토어 노출률 ↑)
- 변호사 검토 시간도 확보

**옵션 B — 클로즈드 베타만 12월에 시작 + 정식 출시 1월**
- "출시"의 정의를 분리:
  - 클로즈드 베타: 12월 후반 (5명, URL 비공개)
  - 정식 출시: 2027년 1월 말
- 베타 결과 반영하고 본 출시 → 더 안전

**옵션 C — 강행 (비권장)**
- 12월 후반에 출시하면 첫 사용자 확보 어렵고 회고 데이터도 적음
- 포트폴리오 가치도 1월 말 출시 대비 떨어짐

### 결정 후 다음 단계

옵션 A (또는 B)로 결정. README + PRD에 새 일정 명시. 변경 자체가 "위기 대응"이 아닌 "합리적 의사결정"임을 본인에게 명확히.

## §7. 12월 위기 — 비용 폭증 ($30/day 한도 도달)

### 증상 체크

- [ ] Anthropic Console에서 "Daily limit reached" 알림
- [ ] 베타 사용자가 채팅 입력 시 에러 발생
- [ ] 월 비용이 예상($50)의 2배 이상 ($100+)

### 원인 진단

Phase 1 단계에서 비용 폭증의 흔한 원인:

1. **캐싱 미적용** — 캐시 버전이 잘못 설정되어 매번 full input
2. **무한 루프** — 사용자 채팅이 자동 재시도 (Edge Function 버그)
3. **봇/스팸** — 가입 후 봇이 1초당 10회 호출

### 대응 옵션

**옵션 A — 즉시 차단**
1. Anthropic Console에서 API 키 회전 (rotate)
2. Spend Limit를 $50/day로 임시 상향
3. Edge Function에 rate limit 즉시 적용 (사용자당 일일 50회)

**옵션 B — 원인 분석**
- Anthropic Console의 호출 로그에서 패턴 확인
- 어느 시간대 / 어느 사용자 / 어느 프롬프트가 비용 증가시키는지
- OPERATIONAL_SAFEGUARDS.md의 비용 상한 4단계 방어 점검

**옵션 C — 캐싱 재점검**
- prompt_caching의 ttl이 실제로 적용되는지 확인
- 시스템 프롬프트 + tool 정의가 캐시 히트하는지 확인
- 정상 시 input 토큰의 95% 캐싱 적중 → 49% 비용 절감

### 결정 후 다음 단계

옵션 A를 즉시 (5분 내). 그 후 B → C 순서로 분석.

## §8. 1월 위기 — 출시 후 사용자 0명

### 증상 체크

- [ ] 출시 1주일 후 신규 가입자 0명
- [ ] 베타 5명 외 활성 사용자 0명
- [ ] 앱스토어 검색에서 본 앱 안 나옴

### 원인 진단

대부분의 1인 출시 앱이 거치는 단계. 정상적인 결과지만 대응 필요.

1. **앱스토어 SEO** — "일본어 학습", "JLPT" 키워드에서 본 앱이 안 나옴
2. **마케팅 부재** — 5월 PRD에 "마케팅 별도"로 분리됐던 것이 실제 발효
3. **타겟 오해** — 본 앱은 1.5만~3만 명 시장, 일반 일본어 학습자에 검색 안 노출

### 대응 옵션

**옵션 A — 일한모 + 페이스북 커뮤니티 알림**
- 일본한국인모임에 출시 알림 글
- 페이스북 일본 거주 한국인 그룹 5-10개에 동시 알림
- 인스타그램에 #일본거주 #일본생활 해시태그로 본인 일상 + 앱 소개

**옵션 B — 베타 5명을 인플루언서로**
- 베타 사용자에게 "리뷰 부탁" — 솔직한 후기 (좋든 나쁘든)
- 그들이 본인 SNS에 공유 (제한적이나 진정성 있음)

**옵션 C — 정직한 회고**
- 1.5만~3만 명 시장에서 1주일 0명은 정상
- 1개월 0명도 가능
- "포트폴리오 검증 무대"의 본질을 다시 확인 — **사용자 수가 핵심 KPI 아님**
- 회고 문서 작성 → 음악 앱 프로젝트로 이전

### 결정 후 다음 단계

옵션 A는 1일 작업. C는 1주일 후 정리. B는 베타 사용자 동의 받고.

## §9. 항상 — 본업 부담으로 작업 중단

### 증상 체크

- [ ] 본업에서 야근/주말 출근이 2주 연속
- [ ] 주말 4시간 작업 약속이 1시간으로 줄어듦
- [ ] 토요일에 작업 못 하고 일요일만 가능

### 원인 진단

본업이 1인 사이드 프로젝트의 가장 큰 위협. 12월 출시 일정과 충돌 시 치명적.

### 대응 옵션

**옵션 A — 일정 후행 보정**
- 1주일 본업 부담 → 1주일 일정 후행
- README의 마일스톤을 즉시 업데이트
- 본인을 비난하지 말고 사실 기록

**옵션 B — 우선순위 재조정**
- 핵심 기능만 남기고 부가 기능 v2로
- 예: 캘린더 탭 출시 후 추가, 학습 지도 v2 그대로

**옵션 C — 출시 시점 미루기**
- §6 옵션과 동일

### 결정 후 다음 단계

본업이 진정될 때까지 일정만 늦추고 작업 자체는 멈추지 말 것. 1주일 100% 멈춤보다 매일 30분 작업이 동기 유지에 유리.

## §10. 보편 원칙 — 위기 시 회피해야 할 함정

위기 대응에서 본인이 빠지기 쉬운 함정 5개:

### 1. "조금만 더 하면 끝나" 함정
- 증상: 마감일 1주일 전에 작업이 끝나지 않음
- 위험: "야근으로 끝내자" → 품질 저하 + 번아웃
- 대응: 끝낼 수 없는 양이라면 **즉시 일정 변경**. "주말 강행"보다 "1주일 미루기"가 항상 옳음.

### 2. "그래도 일단 출시" 함정
- 증상: 핵심 기능이 안 끝났는데 출시 강행
- 위험: 첫 사용자 경험이 망가지면 회복 불가능
- 대응: **출시는 한 번뿐**. 미루는 비용 < 망친 출시의 비용.

### 3. "혼자 해결" 함정
- 증상: 막힌 문제를 1주일 혼자 해결 시도
- 위험: 시간 낭비 + 동기 저하
- 대응: 4시간 막히면 **AI 또는 사람에게 질문**. 시간이 가장 비싼 자원.

### 4. "완벽주의" 함정
- 증상: 첫 초안이 출판 수준이어야 함
- 위험: 진도 안 나감 + 평가 두려움
- 대응: **나쁜 초안 → 좋은 수정**. 첫 작성은 항상 나쁘다.

### 5. "사용자 없으니 의미 없어" 함정
- 증상: 출시 후 1주일 0명 → "이거 의미 있나" 의문
- 위험: 이 시점에 멈추면 **포트폴리오 가치 제로**
- 대응: 본 프로젝트의 핵심 가치는 **AI 디자인 프로세스 검증**. 사용자 0명도 그 가치는 검증됨. README 다시 읽기.
